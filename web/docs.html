Installation

Postier is a desktop application built with Wails (Go + React). Pre-built packages are available for all major platforms.

Download a release

Head to the GitHub releases page and grab the package for your platform:

  • Linux — .deb, .rpm, .apk
  • AUR — postier-bin
  • Windows — .exe installer
  • macOS — .dmg

Build from source

Prerequisites: Go 1.23+, Node.js with npm, and the Wails CLI.

# Install Wails CLI
go install github.com/wailsapp/wails/v2/cmd/wails@latest

# Clone the repo
git clone https://github.com/results-may-vary-org/postier.git
cd postier

# Install frontend dependencies
npm install --prefix frontend

# Build
wails build

For local development with hot-reload, use wails dev instead of wails build.

Quick Start

Launch Postier. You'll see a split pane: the left sidebar for your collections and the main area for composing requests.

  1. Click New File in the sidebar to create a request file.
  2. Select a method (GET, POST, etc.) and enter a URL.
  3. Add headers, query params or a body if needed.
  4. Press Send or hit Ctrl+Enter.
  5. Inspect the response in the tabs on the right: Body, Headers, Cookies, Request, Timeline.

Requests are auto-saved by default when you hit send. You can enable / disable that on the settings.

Collections

Collections are simply local folders on your filesystem. Postier imports them as-is — no sync, no cloud, no lock-in.

Adding a collection

Click the + Collection button in the sidebar and select a folder. Postier will display all .postier files in that folder tree.

Organising requests

Create, rename, and delete files and folders directly from the sidebar tree. The currently open file is highlighted. Requests are stored as plain JSON — commit them to git like any other project file.

Collections are great for sharing API examples with teammates. Just commit your .postier files to git.

Request Builder

The request builder gives you full control over every aspect of an HTTP request.

Method & URL

Select a method from the dropdown: GET, POST, PUT, DELETE, PATCH, HEAD, OPTIONS. Enter the full URL including scheme. Query parameters can be added inline or via the Params tab.

Headers

Add any number of request headers as key-value pairs. Common headers like Content-Type are set automatically based on the body type you select.

Query Parameters

Use the Params tab to manage query string parameters as structured key-value pairs. They sync with the URL in real time at send time.

Body Types

Postier supports the following request body types:

Type Content-Type Use case
JSON application/json REST APIs, GraphQL
XML application/xml SOAP, legacy APIs
Text text/plain Plain text payloads
SPARQL application/sparql-query Semantic web / RDF endpoints
raw you need to set it Send a body type that doesn't exist yet in postier
None no body GET, HEAD, DELETE requests

Environment Variables

Postier supports environment variables via a .postier.env file placed at the root of your collection folder.

Syntax

Use {{VARIABLE_NAME}} anywhere in URL, headers, query params, or body. Variables are resolved at send time. 

// .postier.env (JSON file)
{
  "BASE_URL": "https://api.example.com",
  "API_KEY":  "sk-live-...",
  "USER_ID":  "42"
}

Then in your request URL:

{{BASE_URL}}/users/{{USER_ID}}/posts

Secret Masking

Any key whose name contains key, token, secret, password, pass, auth, credential, or private (case-insensitive) is automatically masked in the environment modal UI with a reveal toggle.

Examples: API_KEY, ACCESS_TOKEN, DB_PASSWORD, CLIENT_SECRET.

Masking is purely visual — the value is still sent in the request as-is. It just prevents shoulder-surfing or accidental screenshots.

Redirects

A toggle in the settings lets you control whether Postier follows 3xx redirects. When disabled, you see the raw redirect response. When enabled, all redirect hops appear in the timing breakdown of the Timeline tab.

Response Viewer

After sending a request, the response area shows five tabs:

  • Body — Response body with Raw / JSON toggle and syntax highlighting. Click any JSON path in the explore modal to copy it as an inject reference ({{@filename|body.path}}).
  • Headers — All response headers as a structured list.
  • Cookies — Cookies set by the response.
  • Request — The exact request Postier sent, formatted as HTTP, cURL, wget, or HTTPie.
  • Timeline — Full HTTP trace log. See below.

At the top of the response area: HTTP status code (color-coded), total time, and response size. Timing breakdown per phase (DNS, TCP, TLS, TTFB, Transfer) is visible alongside the timeline while clicking on the request time.

HTTP Trace & Timeline

The Timeline tab shows a chronological log of every step in the HTTP connection. Each line is timestamped to the millisecond.

Logged events include:

  • DNS lookup (resolved address)
  • TCP connection established
  • TLS handshake with protocol version, cipher suite and certificate details
  • The request line and method sent
  • First byte received (TTFB)
  • Full transfer complete
  • Each redirect hop (if redirects are enabled)

HTTP methods are color-coded in the timeline. Status codes are colored by class: 2xx green, 3xx blue, 4xx orange, 5xx red.

[10:24:01.012Z] DNS lookup: api.example.com → 93.184.216.34
[10:24:01.034Z] TCP connected
[10:24:01.058Z] TLS handshake: TLS 1.3, AES_256_GCM_SHA384
[10:24:01.059Z] GET https://api.example.com/users/42
[10:24:01.112Z] HTTP/2.0 200 OK

Export Formats

The Request tab in the response area lets you view and copy the exact request that was sent in four formats:

  • HTTP — Raw wire format (e.g. GET /path HTTP/1.1)
  • cURL — Shell command using curl
  • wget — Shell command using wget
  • HTTPie — http CLI command format

Click the copy button to copy the formatted output to your clipboard.

Run All & Dependency Graph

A run all button is available on the collection root folder to allow you to run all request placed on the collection.

If you have chain some requests together the dependency graph show you the path for each injection.

How it works

Use inject references (see below) to declare that a field in one request depends on a value from another request's response. Postier builds a directed graph from these references.

Click Run all in the collection sidebar. Postier executes requests in dependency order, rendering a live graph as it goes. Independent chains are colored differently so you can see them at a glance.

A complete demo is available in the graph-example/ folder — it uses the Open Brewery DB public API and needs no auth.

Inject Syntax

Use {{@filename|source}} anywhere in a request field to inject a value from another request's response.

Source What it injects Example
body The full raw response body {{@login|body}}
body.path A field from a JSON body (dot notation) {{@login|body.token}}
body[0].field A field from a JSON array body {{@list|body[0].id}}
headers.Name A response header value {{@auth|headers.X-Token[0]}}
status The HTTP status code as a string {{@check|status}}

body.path traversal requires the source response to be valid JSON. body alone works with any content type. 

// In get-user.postier — depends on login.postier
{
  "url": "{{BASE_URL}}/users/{{@login|body.userId}}",
  "headers": {
    "Authorization": "Bearer {{@login|body.token}}"
  }
}

.postier File Format

Each request is stored as a .postier JSON file. The format is stable and human-readable: 

{
  "method": "POST",
  "url": "{{BASE_URL}}/users",
  "headers": {
    "Content-Type": "application/json",
    "Authorization": "Bearer {{API_KEY}}"
  },
  "params": {},
  "bodyType": "json",
  "body": "{\n  \"name\": \"Ada Lovelace\"\n}",
  "followRedirects": true
}

The response is also save in the .postier file, this could be useful to check data while being out-of-connection or use in agent directly from the file.

.postier.env File Format

A flat JSON object placed at the collection root. Keys are variable names, values are strings. 

{
  "BASE_URL":    "https://api.example.com/v2",
  "API_KEY":     "sk-...",
  "TIMEOUT":     "30"
}

Add .postier.env to your .gitignore to avoid committing real secrets. Commit a .postier.env.example instead.
